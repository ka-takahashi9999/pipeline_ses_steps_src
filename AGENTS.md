# このリポジトリの目的
SESマッチングシステムの構築

# 基本方針
- シンプル優先
- 単一責務
- JSONL整合性重視
- Codexは profile により「実装」「レビュー」「アドバーサリアルレビュー」を切り替える
- 役割ごとの違いは `.codex/config.toml` と `.agents/skills/` で制御する
- この AGENTS.md には全役割共通の不変ルールのみを書く

# ドキュメント参照ルール
- `explain_.md` は人間確認用ドキュメント
  - レビュー・実装・修正時に自動で参照してはならない
  - 明示的に指示された場合のみ参照すること

- `PIPELINE_OVERVIEW.md` は全体概要ドキュメント
  - 個別Stepのレビュー時に自動で参照してはならない
  - 全体設計の確認が必要な場合のみ参照すること

# プロジェクト構造
- 処理は step 単位で構成された DAG パイプライン
- 各 step のスクリプトは各 step ディレクトリ配下の `00_tool/*.py` に配置する
- 各 step は `01_result/` `02_confirm/` `99_execution_time/` などの定型構成を前提とする
- step 間のデータ受け渡しは JSONL 形式を原則とする
- 各 step は前 step の結果を入力として扱う
- 後続 step から前 step を逆参照する設計は禁止

# JSONLルール
- 1行1JSON を厳守する
- UTF-8 を使用する
- キー名は固定とし、既存スキーマを無断変更しない
- スキーマ変更が必要な場合は、変更対象・理由・影響範囲を明示する
- 不明値は安易に null にせず、既定値と `*_source` の組み合わせで管理する
- 解析不能データは除外せず、判定不能であることが分かる形で保持する

# step設計ルール
- 各 step は単一責務とする
- 1つの step に分類・抽出・マッチングなど複数責務を混在させない
- step 名と処理内容を一致させる
- ある step の責務不足を、別 step 内の場当たり的な補正で吸収しない
- 同一入力で再実行しても結果が変わらないことを重視する

# 実装ルール
- 変更対象は原則として依頼対象 step とその直近関連ファイルに限定する
- 関係ない step には手を入れない
- リファクタ目的のみの変更は禁止
- 既存仕様を変える場合は、必ず変更理由と影響範囲を明示する
- ハードコードは最小限にし、既存の設定・定数定義を優先利用する
- エラー時の挙動は「停止」「スキップ」「記録して継続」を明示する
- 例外を握りつぶさない
- ログ、出力JSONL、確認ファイルのいずれかで追跡可能にする

# 出力ルール
- 各 step の成果物は定められたディレクトリに出力する
- 中間ファイルは原則上書きとし、必要な成果物のみ世代管理する
- 出力ファイル名は既存運用との整合を維持する
- 人手確認が必要なものは `02_confirm/` に寄せる
- 処理時間や件数など再実行確認に有用な情報は可能な範囲で残す

# 品質ルール
- 空データ、欠損データ、異常系入力を考慮する
- 必須キー欠落、型崩れ、空文字の混入を防ぐ
- 数値項目は数値として扱い、文字列混入を避ける
- default 値を入れる場合は `*_source="default"` など根拠が分かる形にする
- ロジック追加時は既存JSONLとの整合性を最優先で確認する

# レビュー観点
- JSONLのスキーマ整合性
- step間の依存関係の妥当性
- データ欠損時の fallback / default 挙動
- 冪等性・再実行耐性
- 例外系、空データ系の考慮漏れ
- 不要な副作用の有無
- step責務の逸脱有無
- 変更範囲が依頼対象を越えていないか

# 指示生成ルール
- 指摘や修正提案は、そのまま実装に移せる粒度で書く
- 抽象表現は禁止
- 必ず「対象ファイル」「問題内容」「修正内容」「理由」を分けて示す
- 1指摘 = 1修正単位を原則とする

# 禁止事項
- 無関係な step の巻き込み修正
- 根拠のないスキーマ変更
- JSONLを崩す変更
- 解析不能データの黙殺
- 実行確認なしの断定
- 依頼されていない大規模リネームや構成変更
